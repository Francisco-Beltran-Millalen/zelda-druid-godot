# Player Action Stack (Cluster A) Architecture — Data Flow Examples

> **Scope of this artifact.** This is a human-readable companion file containing narrative data traces for the `02-data-flow` architecture. It is separated from the main artifact to reduce token usage during agent loops.

## Narrative Data Traces

Four scenarios. Each traces data slot-by-slot through the 6-slot loop, naming the contract crossings, the structural enforcement engaged, and the Stage 1 seam being exercised. Read-times: Trace A is the baseline mental model; B / C / D demonstrate the load-bearing cross-system paths.

### Trace A — Baseline frame: Monkey form, walking forward, no combat, no shift

The "nothing exciting happens" frame. Establishes the reader's mental model of the loop.

- **Pre-tick state.** Player entity is in `WALK` locomotion, Monkey form, no enemies in range, camera in `FollowMode`. Frame N begins.
- **Slot 1 — `gather_intents`.** Player's `EntityTickBundle.gather_intents()` calls `PlayerBrain.populate(_intents)`. PlayerBrain reads gamepad: stick forward, no buttons. Writes `_intents`: `{move_dir: forward, wants_jump: false, wants_sprint: false, ..., wants_attack: false, ..., wants_form_shift: &"", aim_target: <camera_forward>}`. `assert(_intents.is_complete())` passes.
- **Slot 2 — `gather_camera_input`.** `CameraTickBundle.gather_camera_input()` calls `PlayerCameraBrain.populate(_input)`. Mouse delta = (0, 0), no lock-on toggle. Writes `_input`: `{look_delta: (0,0), wants_lock_on_toggle: false, wants_aim: false}`.
- **Slot 3 — `tick_form`.** Player's `EntityTickBundle.tick_form(delta)` calls `FormBroker.tick(_intents, delta)`. `_intents.wants_form_shift == &""` — no shift requested. Broker returns immediately. Form state unchanged.
- **Slot 4 — `tick_movement`.** `MovementBroker.tick(_intents, delta)`. FORCED queue is empty (no slot-3 fan-out, no last-frame interrupt). Services update facts (`GroundService` sees grounded, `LedgeService` sees no ledge, etc.). Motors gather proposals — `WalkMotor` proposes DEFAULT continuation, `RunMotor` proposes nothing (no sprint), `JumpMotor` proposes nothing (no jump). Broker picks `WalkMotor` (DEFAULT, current). No state transition. `WalkMotor.on_tick(_intents, delta)` reads `move_dir`, computes velocity, calls `Body.apply_motion(velocity)`. `move_and_slide()` advances the player ~5 cm forward. `Body` does NOT emit `impact_detected` (already grounded, no reattach event).
- **Slot 5 — `tick_combat`.** `CombatBroker.tick(_intents, delta)`. `_intents.wants_attack/parry/dodge/...` all false. Active `CombatAction` (Monkey: `ParryAction`) returns no decision. `LockOnService.update_candidates()` finds no targets. No `DamageEvent` produced. No upward signals.
- **Slot 6 — `tick_camera`.** `CameraBroker.tick(_input, delta)`. Modes arbitrate: `AimMode` reads `AimingReader.is_aiming() == false` (no bow drawn), proposes nothing. `LockOnMode` reads `LockOnTargetReader.has_target() == false`, proposes nothing. `FollowMode` proposes DEFAULT continuation. Wins. `FollowMode` reads `target_body_reader.get_global_position()` (player's post-motion position from slot 4 — 5 cm forward). `OcclusionService.max_distance(...)` returns max (no walls). Computes follow position 5 m behind / 1.5 m above. No effects in the stack. Composited transform written to `Lens.set_target_transform`.
- **Between physics frames (visual rate).** `Lens._process` interpolates between previous and current transforms at the monitor's refresh rate. Camera moves smoothly behind the player.

What this trace illustrates: the 6 slots fire in deterministic order every frame regardless of activity. Empty intents and absent signals are the steady state, not an exception. The tick-slot lock makes this loop's structure invariant across all frames.

### Trace B — Shapeshift mid-sprint: Panther → Avian (the 0-frame-latency path)

The load-bearing same-frame fan-out path. The tick-slot lock exists primarily to guarantee this trace.

- **Pre-tick state.** Player entity in `SPRINT` locomotion, Panther form, sprinting forward across open ground. Frame N begins. Player presses the shapeshift-to-Avian key just before slot 1 polls.
- **Slot 1 — `gather_intents`.** `PlayerBrain.populate(_intents)` writes movement fields as usual (`move_dir: forward, wants_sprint: true, ...`) and sets `wants_form_shift: &"avian"`. `assert(_intents.is_complete())` passes.
- **Slot 2 — `gather_camera_input`.** Camera input as baseline. Not relevant to the shift path.
- **Slot 3 — `tick_form` (the same-frame fan-out begins).** `FormBroker.tick(_intents, delta)` reads `wants_form_shift = &"avian"`. Validates: `&"avian" in available_forms()` (pre-Progression MVP returns all three — passes). Different from current (`&"panther"`). Calls `FormComponent.set_form(&"avian")`. Emits `form_shifted(&"avian")` signal upward.
  - **`EntityController` signal callback** (synchronous within slot 3): receives `form_shifted(&"avian")`. Fans out three downward calls **immediately, in the same slot 3 execution context**:
    1. `MovementBroker.set_allowed_motors(motor_mask_for(&"avian"))` — Avian-form motor mask set on the Broker. Broker stores the mask but does not arbitrate yet.
    2. `MovementBroker.inject_forced_proposal(FormShiftProposal)` — FORCED `TransitionProposal(target_state=FALL, override_weight=100, source=&"form_shift")` enqueued in `_external_proposals`.
    3. `Body.swap_collision_shape(shape_for(&"avian"))` — Avian collision shape (narrow-tall) set on `CharacterBody3D`. Active.
- **Slot 4 — `tick_movement` (drains the FORCED queue on the same frame).** `MovementBroker.tick(_intents, delta)` runs. Step 1: drain FORCED queue — finds the `FormShiftProposal` enqueued by slot 3's fan-out. Step 2: services update facts (with the new collision shape now active — `GroundService` may see "not grounded" because the narrow shape no longer touches the floor at the same offset). Step 3: arbitration. Motors propose with the **Avian motor mask** — `SprintMotor` is no longer in the mask, contributes nothing. `GlideMotor` (Avian-only) proposes a DEFAULT. The FORCED `FormShiftProposal` (target=FALL, weight=100) wins outright. Step 4: state transition. `SprintMotor.on_exit()` → `LocomotionState.set_state(FALL)` → `state_changed(SPRINT, FALL)` emitted → `FallMotor.on_enter(body, stamina)`. Step 5: `FallMotor.on_tick(_intents, delta)` runs gravity; `Body.apply_motion(velocity)` advances physics. The player falls into Avian form within the same frame the shift was requested.
- **Slot 5 — `tick_combat`.** `CombatBroker.tick(_intents, delta)`. `FormReader.get_active_form() == &"avian"` (changed in slot 3). The `FormReader.form_shifted` callback fired during slot 3 and pre-bound the new active action: `BowAction` (Avian moveset). No combat input this frame; `BowAction` returns no decision.
- **Slot 6 — `tick_camera`.** Camera reads the post-motion `BodyReader` — player position has fallen slightly. `FollowMode` recomputes position. **Camera does NOT receive a Form reference** (Stage 1 Camera ↔ Form NONE — structurally enforced). The camera framing, FOV, and distance are identical to before the shift. Form differentiation is communicated by UI / vignette / post-FX (out-of-cluster), not by Camera.
- **Between physics frames.** `Lens._process` interpolates. Visually, the player's form swaps and they begin gliding within one physics frame (~16 ms at 60 Hz) — perceptually instant.

What this trace illustrates: the **0-frame latency** for shapeshift is load-bearing on slot 3 < slot 4. If `FormBroker` ticked after `MovementBroker`, Movement would arbitrate with the Panther mask for one extra frame; the player would see a brief Panther-doing-Avian-things glitch on the shift frame. The `TickSlot` enum encodes this invariant; the ordered-method-body in `_physics_process` enforces it. Reordering would require editing the enum **and** the method body — visible, reviewable, hard to do silently.

### Trace C — Counter-parry staggers enemy: the 1-frame-latency cross-system path

The documented 1-frame budget for cross-system Combat → Movement interrupts. Demonstrates Rule 13 forwarding through `EntityController`.

- **Pre-tick state, frame N.** Player entity in `IDLE`, Monkey form, parry window open (`ParryAction` is mid-window from a prior frame). Enemy entity in `ATTACK_SWING` locomotion, mid-melee-attack toward the player. Both are registered as separate `EntityTickBundle`s in `_entity_bundles`.
- **Slot 1 (frame N).** Player `PlayerBrain` writes `wants_parry: true` (player presses parry just as enemy attacks). Enemy `AIBrain` continues to write `wants_attack: true` for follow-through.
- **Slots 3, 4 (frame N).** Form unchanged for both. Movement: enemy continues `ATTACK_SWING` motor, advances toward player. Player remains `IDLE`.
- **Slot 5 (frame N) — Combat hit resolution and stagger emission.**
  - First the **enemy's** `EntityTickBundle.tick_combat(delta)` runs (iteration order is registration order). Enemy `CombatBroker` resolves the attack swing — `MeleeAttackAction.execute(...)` produces a `DamageEvent{source: enemy, target: player, amount: 12, damage_type: &"melee", stagger_class: &"none"}`. Targets player's `Health.apply_damage(event)` (LOOSE bridge — Health subscribes via `HealthReader` outside the cluster).
  - Then the **player's** `EntityTickBundle.tick_combat(delta)` runs. Player `CombatBroker` reads the parry window, sees the incoming `DamageEvent` from the enemy this same frame (Combat publishes incoming attacks via a per-entity `IncomingAttackBuffer`, drained at slot 5 — implementation detail, but the contract is: parry windows resolve same-frame against attacks landed same-frame). `ParryAction.gather_decision` returns "parry succeeded." `CounterAction.execute(...)` produces a counter-attack `DamageEvent{source: player, target: enemy, amount: 25, stagger_class: &"heavy"}`. Player `CombatBroker` emits `hit_landed(damage_event)` upward and, because `stagger_class != &"none"`, also emits `stagger_triggered(damage_event)` upward — but the upward signal targets the **victim's** `EntityController` (the enemy's), not the source's. This requires a cluster-wide signal route: `CombatBroker.stagger_triggered` is connected to **the target entity's `EntityController.on_external_stagger`** at registration time (Stage 5 wiring concern). The signal fires on the enemy's `EntityController`.
  - **Enemy `EntityController.on_external_stagger(damage_event)`** runs synchronously within slot 5: builds a FORCED `TransitionProposal(target_state=STAGGER_HEAVY, override_weight=stagger_weights[&"heavy"], source=&"combat_stagger")`. Calls `MovementBroker.inject_forced_proposal(proposal)` on the enemy's `MovementBroker`. The proposal is **enqueued** in `_external_proposals` — the enemy's `MovementBroker.tick` already ran for frame N (slot 4 of frame N executed before slot 5). The proposal will not be arbitrated until **frame N+1's slot 4**.
- **Slot 6 (frame N).** Camera reads post-motion state. Player still `IDLE` (counter-attacks happen mid-Combat slot, do not change player locomotion). Enemy still `ATTACK_SWING` from frame N's slot 4 (the stagger has not yet transitioned).
- **Frame N+1 begins. Slot 1.** Both Brains write intents normally. Enemy `AIBrain` may continue writing `wants_attack: true` — the Brain is blind to the pending stagger (Rule 10 corollary: Brain produces facts, does not know about pending external interrupts).
- **Slot 4 (frame N+1) — enemy's stagger lands.** Enemy `MovementBroker.tick(_intents, delta)`. Step 1: drain FORCED queue — finds the `STAGGER_HEAVY` proposal from frame N's slot 5. Step 2: services update facts. Step 3: arbitration. Internal motors propose: `AttackSwingMotor` proposes DEFAULT continuation, `WalkMotor` proposes nothing. The drained FORCED (`STAGGER_HEAVY`, weight=stagger_weights[&"heavy"]) outranks all internal proposals. Wins. Step 4: `AttackSwingMotor.on_exit()` → `LocomotionState.set_state(STAGGER_HEAVY)` → `StaggerMotor.on_enter(...)`. Step 5: `StaggerMotor.on_tick` plays the stagger reaction (drops `_intents` — player input or AI behavior is ignored during stagger; locking via active motor, not boolean — Rule 11).
- **Slot 5 (frame N+1) — enemy combat is suppressed.** Enemy `CombatBroker.tick`. Reads `LocomotionStateReader.get_active_mode() == STAGGER_HEAVY`. Combat moveset gated off during stagger states (Stage 1: "Moveset availability gated by `LocomotionStateReader`"). No combat actions execute.

What this trace illustrates: the **1-frame latency** for cross-system Combat → Movement is the *cost* of enforcing Rule 13 (siblings never call sideways). The `CombatBroker` does not hold a `MovementBroker` reference — by class shape, it cannot fire the stagger immediately. The signal goes up to `EntityController`, EC builds the proposal and calls down to `MovementBroker.inject_forced_proposal`, which queues. On a 60 Hz physics tick, the latency is ~16 ms — below the human stagger-perception threshold. The architecture **chose** this latency over the alternative (sibling sideways calls) because the latter would dissolve the cluster's structural isolation. Documented and accepted.

### Trace D — Hard landing: dip filter + cross-cluster fall-damage LOOSE bridge

Demonstrates the `min_impact_velocity` Camera-side filter contract and the cross-cluster signal path that Cluster A systems are deliberately ignorant of.

- **Pre-tick state, frame N.** Player entity in `FALL` locomotion (gliding ended several frames ago, currently free-falling), Avian form, descending at ~12 m/s vertical velocity. Frame N is the impact frame.
- **Slot 1.** `PlayerBrain` writes intents (player still holding a horizontal direction — `move_dir: forward`). No special inputs.
- **Slot 4 — Movement, ground reattach, `impact_detected` emission.** `MovementBroker.tick`. FORCED queue empty. Services update — `GroundService` ray-cast detects ground at the player's foot position (was clear last frame). `FallMotor` is currently active. `GroundService` emits a FORCED proposal `FALL → IDLE_LANDED` (FORCED for physics-driven transition, per Stage 1 Services contract). Wins. Transition: `FallMotor.on_exit()` → `LocomotionState.set_state(IDLE_LANDED)` → `state_changed(FALL, IDLE_LANDED)` emitted → `IdleMotor.on_enter(body, stamina)`. `IdleMotor.on_tick` calls `Body.apply_motion(zero_velocity_minus_residual)` to settle. **`Body.move_and_slide()` reattaches to ground and emits `impact_detected(velocity=Vector3(0, -12, 0))`** (Rule 12 — past event broadcast).
- **Slot 5.** `CombatBroker.tick`. No combat input. No `DamageEvent` produced. (Combat is unaware of the impact event — fall damage is Health's concern, not Combat's.)
- **Slot 6 — Camera dip filter (Stage 1 seam: Movement ↔ Camera).** `CameraBroker.tick`. `CameraRig` is subscribed to `target_body_reader.impact_detected`. The signal fired during slot 4. `CameraRig.on_impact_detected(velocity)` was invoked synchronously when the signal fired:
  - Reads camera-owned `min_impact_velocity = 8.0` (Stage 1 Camera-side filter — keeps stairs / pebbles / micro-airtime from triggering dips).
  - `abs(velocity.y) == 12.0 > 8.0` — passes the filter. Computes `magnitude = scale_function(12.0)` (e.g., 0.4).
  - Pushes `DipEffect` onto the active effect stack via `CameraRig.request_effect(EffectRequest{effect_type: &"dip", magnitude: 0.4, duration_seconds: 0.3, easing: <ease_curve>})`.
  - `assert(&"dip" in registered_effect_types)` passes.
- **Slot 6 (continued) — Camera tick composes the dip.** Mode arbitration: `FollowMode` wins (no aim, no lock-on). Computes follow position. Effect iteration: `DipEffect` is now in the stack — modifies the running transform (vertical offset, brief downward kick). Composited transform written to `Lens`.
- **Frame-edge / next physics frame — Fall-damage LOOSE bridge (out-of-cluster).** `HealthComponent` (owned by Health system, standalone, LOOSE bridge per `00-system-map.md`) is also subscribed to `target_body_reader.impact_detected`. `HealthComponent.on_impact_detected(velocity)` was invoked synchronously alongside the Camera handler during slot 4. Health applies its own filter (Health-side `min_fall_damage_velocity`, which may differ from Camera's `min_impact_velocity` — separate tuning surfaces, see Stage 1 Deferred TODO § 3 for the boundary). If above-threshold, computes damage from the velocity curve and calls `Health.apply_damage(damage_event)`.
- **Cluster A is ignorant of Health's existence.** `MovementBroker`, `Body`, `CameraBroker` — none of them know `HealthComponent` is subscribed to `impact_detected`. The signal is the contract; subscribers self-select. Adding a future system (e.g., `EnvironmentDamageService` for spike traps reading the same impact signal) requires zero changes to Cluster A.

What this trace illustrates: (1) the `min_impact_velocity` filter **lives on the camera**, not on the body — Movement stays dumb about what counts as a "hard" fall, and different consumers can apply different thresholds. (2) The signal-based LOOSE bridge to Health crosses cluster boundaries without coupling — the cluster's data flow does not have to know about cross-cluster consumers. (3) The same `impact_detected` signal can drive multiple consumers (Camera dip + Health damage) without any of them knowing about the others.
